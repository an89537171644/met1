# Первые GitHub Issues

Ниже задачи, которые удобно создать после публикации репозитория.

## 1. Sprint 0: реализовать Pydantic-модели входных данных

Цель: создать базовые модели `FrameGeometry`, `SteelMaterial`, `SectionProfile`, `SP20LoadCase`, `LoadCombination`, `LiraImportRecord`, `MLCase`.

Контекст:

- `docs/TZ.md`
- `docs/sp20-loads.md`
- `docs/ml-dataset.md`
- `config.example.yaml`

Критерии готовности:

- модели лежат в `src/steel_frame_designer/`;
- есть тесты валидации геометрии;
- `pytest` проходит;
- ошибки валидации понятны пользователю.

## 2. Sprint 0: реализовать загрузку и проверку `config.example.yaml`

Цель: добавить загрузку YAML-конфига и проверку обязательных полей.

Критерии готовности:

- есть модуль `config.py`;
- CLI-команда `validate-input` читает YAML;
- пустые обязательные поля показываются в отчете ошибок;
- `pytest` проходит.

## 3. Sprint 1: геометрия двухскатной рамы

Цель: вычислять узлы, элементы, длины элементов и уклон кровли.

Критерии готовности:

- для схемы N1-N5/E1-E4 строится объект рамы;
- уклон вычисляется из пролета и высот;
- некорректные высоты и пролеты отклоняются;
- есть тесты.

## 4. Sprint 2: структура нагрузок по СП 20

Цель: реализовать структуры данных для постоянных, снеговых, ветровых и ремонтных нагрузок без зашитых нормативных коэффициентов.

Критерии готовности:

- нагрузки имеют `load_id`, группу, схему, направление, единицы, грузовую ширину, источник нормы;
- коэффициенты берутся из конфигурации/таблицы;
- есть валидация единиц и пустых значений.

## 5. Sprint 3: валидатор CSV-экспортов ЛИРА

Цель: проверить текущие CSV-шаблоны и подготовить понятный отчет о недостающих данных.

Критерии готовности:

- проверяются обязательные файлы;
- проверяются связи `case_id`, `combo_id`, `section_id`;
- отчет явно показывает пустые поля и противоречия;
- текущие демо-шаблоны распознаются как неполные.
