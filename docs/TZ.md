# Техническое задание v0.2
## Проект: программа подбора единого сечения для плоских двухскатных однопролётных стальных рам

**Контекст:** магистерский проект Станислава.  
**Дата обновления:** 2026-06-07.  
**Базовая нормативная постановка:** СП 16.13330.2017 «Стальные конструкции» для проверок стальных элементов; СП 20.13330.2016 «Нагрузки и воздействия» для нагрузок, воздействий, сочетаний, прогибов и перемещений. В проекте обязательно хранить редакцию СП и номер изменения, потому что для СП 16 и СП 20 есть изменения.

---

## 1. Принятые уточнения

1. Расчёт стальных конструкций выполняется по **СП 16.13330.2017**.
2. Нагрузки и сочетания выполняются по **СП 20.13330.2016**.
3. На первом этапе рассматриваются **плоские двухскатные однопролётные рамы**.
4. Сечение должно быть **одно общее** для основных элементов рамы: стоек и двух наклонных ригелей/стропильных частей.
5. Связи, прогоны, фахверк, узлы, фундаменты и ограждающие конструкции в MVP не подбираются, но их веса могут участвовать в постоянной нагрузке.
6. Расчётная обучающая база формируется на основе расчётов в **ЛИРА-САПР / ПК ЛИРА**.
7. Нейронная сеть должна подбирать или рекомендовать сечение, но итоговое проектное решение должно проходить проверку расчётным модулем или ЛИРА.
8. Нужен пользовательский интерфейс. Для MVP рекомендован CLI + Streamlit-интерфейс; полноценный web-интерфейс можно оставить на второй этап.
9. Форматы отчётов выбираются так: Markdown + Excel/CSV в MVP, PDF — после стабилизации расчётов.

## 2. Назначение программы

Разработать программу, которая по заданным параметрам плоской двухскатной однопролётной стальной рамы автоматически подбирает единое стальное сечение для стоек и ригелей.

Основной сценарий:

```text
геометрия рамы → нагрузки по СП 20 → сочетания по СП 20 → расчётная модель → усилия/перемещения → подбор единого сечения → проверка по СП 16 → отчёт
```

После накопления базы расчётов:

```text
геометрия + нагрузки + условия → нейросеть → прогноз сечения/top-N сечений → обязательная проверка → отчёт
```

## 3. Объект расчёта

### 3.1. Тип рамы

- Плоская рама.
- Однопролётная.
- Двухскатная.
- Одноэтажная.
- Расчёт в плоскости поперечной рамы.

### 3.2. Геометрическая схема

Базовая схема включает 5 узлов:

```text
N1 — левая опора
N2 — верх левой стойки / карниз
N3 — конёк
N4 — верх правой стойки / карниз
N5 — правая опора
```

Базовые элементы:

```text
E1: левая стойка N1-N2
E2: левый наклонный ригель N2-N3
E3: правый наклонный ригель N3-N4
E4: правая стойка N4-N5
```

Параметры геометрии:

| Параметр | Обозначение | Единица | Описание |
|---|---:|---:|---|
| Пролёт | L | м | Расстояние между осями стоек |
| Высота до карниза | H_eave | м | Высота от отметки опоры до узла карниза |
| Высота до конька | H_ridge | м | Полная высота до конька |
| Уклон кровли | alpha | град | Может вычисляться из L, H_eave, H_ridge |
| Шаг рам | B | м | Грузовая полоса одной плоской рамы |
| Длина здания | L_building | м | Для ветра, количества рам и объёма данных |
| Количество рам | n_frames | шт | Для массы здания и отчёта |
| Тип опирания | base_fixity | - | Шарнирное/жёсткое/заданное пользователем |

## 4. Основной критерий подбора

Подбирается самое лёгкое сечение из заданного сортамента, которое удовлетворяет всем обязательным проверкам.

Критерий:

```text
min mass(section)
subject to:
  all strength checks pass
  all stability checks pass
  all displacement/deflection checks pass
  section is the same for columns and rafters
```

Если нейросеть выдаёт несколько вариантов, программа должна проверять их в порядке:

1. top-1 прогноз;
2. top-3/top-5 прогнозов;
3. при неуспехе — классический перебор по сортаменту от лёгкого к тяжёлому.

---

## 5. Обязательные расчётные проверки

### 5.1. Для MVP

В MVP допускается использовать результаты ЛИРА как эталонные метки:

- проходит / не проходит сечение;
- максимальный процент использования;
- определяющий элемент;
- определяющее сочетание;
- определяющая проверка.

Программа не должна утверждать, что выполнила полный нормативный расчёт по СП 16, пока модуль проверок не реализован и не верифицирован.

### 5.2. Для инженерного модуля проверок

Рекомендуемый набор проверок:

1. Прочность центрально растянутых/сжатых элементов.
2. Прочность изгибаемых элементов.
3. Прочность элементов при совместном действии продольной силы и изгиба.
4. Проверка устойчивости сжатых элементов.
5. Проверка устойчивости сжато-изгибаемых элементов.
6. Проверка общей устойчивости изгибаемых элементов.
7. Проверка местной устойчивости стенки и поясов.
8. Проверка предельной гибкости элементов.
9. Проверка вертикальных прогибов ригелей.
10. Проверка горизонтального перемещения верха стоек / конька.
11. Проверка реакций опор и корректности расчётной схемы.
12. Проверка коэффициента использования: `max_utilization <= 1.0`.

Узлы, сварные швы, болты, база колонны и фундаменты — отдельный этап после MVP.

## 6. Нагрузки и воздействия по СП 20

Подробный список вынесен в файл `sp20_required_loads_checklist.md`. В программу обязательно заложить структуру, позволяющую хранить не только значение нагрузки, но и её происхождение, единицы, коэффициенты, длительность, сочетания и редакцию СП.

Минимальные обязательные нагрузки для двухскатной рамы:

1. Собственный вес стальной рамы.
2. Постоянная нагрузка от кровельного пирога.
3. Постоянная нагрузка от прогонов, связей, ограждений, подвесов и инженерных систем.
4. Снеговая нагрузка: симметричная и несимметричные схемы для двухскатной кровли.
5. Ветровая нагрузка: направления `+X` и `-X`, давление/отсос по стенам и кровле, внутреннее давление при необходимости.
6. Полезная/ремонтная нагрузка на кровлю, если требуется по назначению.
7. Температурное климатическое воздействие, если рама/здание имеет ограничения деформаций или значительную длину.
8. Гололёдная, крановая, технологическая и особая нагрузки — условно, если они есть в задании.

## 7. Модуль `sp20_loads`

Назначение: формировать нагрузочные случаи и сочетания для плоской рамы по исходным данным пользователя.

Функции модуля:

```python
load_project_location()
load_climate_parameters()
build_dead_load_cases()
build_snow_load_cases()
build_wind_load_cases()
build_roof_live_load_cases()
build_temperature_load_cases()
build_optional_load_cases()
build_load_combinations()
export_load_table()
```

Выход модуля:

```text
load_cases.json
load_combinations.json
load_report.md
```

Требования:

1. Не задавать коэффициенты в коде «магическими числами» без ссылки на норму/редакцию.
2. Все коэффициенты должны храниться в конфигурации или нормативной таблице.
3. Для каждого сочетания сохранять тип предельного состояния: первая или вторая группа.
4. Для каждой нагрузки сохранять направление, единицы, грузовую ширину и способ приложения к раме.
5. Для ветра и снега сохранять схему нагрузки, потому что для двухскатной кровли определяющим может быть несимметричный случай.

## 8. Модуль `lira_import`

Назначение: импорт результатов расчётов из ЛИРА для создания обучающей базы.

Входы:

1. Экспорт таблиц усилий по элементам.
2. Экспорт перемещений узлов.
3. Экспорт реакций опор.
4. Экспорт результатов проверок/процентов использования.
5. Таблица назначенных сечений.
6. Паспорт модели: версия ЛИРА, дата расчёта, используемые нормы, имя файла модели.

Выходы:

```text
data/processed/lira_cases.parquet
data/processed/lira_element_forces.parquet
data/processed/lira_labels.parquet
reports/lira_import_report.md
```

Требования:

1. Поддержать CSV как базовый формат обмена.
2. Позже добавить XLSX/XML/TXT, если такой экспорт удобнее из ЛИРА.
3. Проверять единицы измерения.
4. Проверять, что все модели рассчитаны в одной нормативной логике.
5. Сохранять исходные файлы без изменения.
6. Для каждого расчёта формировать `case_id` и `model_hash`.
7. Не смешивать данные разных редакций СП без отдельного признака `norm_version`.

## 9. Модуль `ml_data`

Назначение: подготовка данных для обучения нейронной сети.

Функции:

```python
load_lira_exports()
normalize_units()
validate_ml_dataset()
build_features()
build_labels()
split_train_val_test()
save_dataset()
make_dataset_card()
```

### 9.1. Рекомендуемые признаки модели

Группа `geometry`:

- пролёт;
- высота карниза;
- высота конька;
- уклон кровли;
- шаг рам;
- длина здания;
- тип опирания.

Группа `loads`:

- постоянная нагрузка на кровлю;
- собственный вес, если задан отдельно;
- снеговой район или нормативное значение;
- ветровой район или нормативное значение;
- тип местности для ветра;
- снеговые схемы;
- ветровые схемы;
- ремонтная/эксплуатационная нагрузка;
- температурные воздействия, если есть.

Группа `analysis_results`, если модель обучается после расчёта:

- максимальные N, Q, M по элементам;
- реакции;
- перемещения;
- длины элементов;
- расчётные длины;
- коэффициенты использования из ЛИРА.

Группа `target`:

- выбранное минимальное сечение;
- масса погонного метра;
- максимальный коэффициент использования;
- определяющий элемент;
- определяющее сочетание;
- pass/fail для каждого проверенного сечения.

### 9.2. Рекомендуемые постановки ML-задачи

**Вариант А — классификация:** нейросеть предсказывает марку сечения из сортамента.  
**Вариант Б — регрессия:** нейросеть предсказывает требуемую массу/площадь/момент сопротивления, затем программа выбирает ближайшее большее сечение.  
**Вариант В — ранжирование:** нейросеть выдаёт top-N кандидатов, а программа проверяет их расчётным модулем.

Рекомендуемый вариант для проекта: **В + обязательная проверка**.

## 10. Модуль `ml_model`

Назначение: обучение, валидация и применение нейросети.

Функции:

```python
train_model()
evaluate_model()
predict_section()
predict_top_k_sections()
explain_prediction()
save_model()
load_model()
```

Метрики:

1. Accuracy top-1 по выбранному сечению.
2. Accuracy top-3/top-5.
3. Среднее отклонение массы выбранного сечения от эталона ЛИРА.
4. Доля небезопасных прогнозов: предсказанное сечение легче эталонного и не проходит проверку.
5. Доля избыточных прогнозов: сечение проходит, но тяжелее эталона.
6. Confusion matrix по классам сечений.
7. Проверка обобщения на новых диапазонах пролётов/нагрузок.

Критическое правило:

> Нейросеть не является нормативным расчётом. Она предлагает сечение, но программа должна проверять результат перед выдачей отчёта.

## 15. Definition of Done для MVP

MVP готов, если:

1. Пользователь может ввести геометрию двухскатной плоской рамы.
2. Пользователь может задать нагрузки по структуре СП 20.
3. Программа может загрузить экспорт ЛИРА.
4. Программа может собрать обучающий датасет.
5. Программа может обучить baseline-модель.
6. Программа может предсказать top-N сечений.
7. Программа может сформировать отчёт.
8. Все ключевые модули имеют тесты.
9. В отчёте ясно указано, что нейросеть является рекомендательной частью, а не заменой нормативной проверки.
