# Спецификация модуля подготовки данных для нейросети

**Дата:** 2026-06-07  
**Проект:** подбор единого сечения двухскатной плоской стальной рамы.  
**Источник эталонных расчётов:** ЛИРА-САПР / ПК ЛИРА.

---

## 1. Цель модуля

Создать воспроизводимый pipeline, который превращает набор расчётов ЛИРА в обучающий датасет для нейросети, подбирающей единое сечение рамы.

```text
экспорты ЛИРА → нормализация → проверка единиц → признаки → метки → train/val/test → baseline модель
```

---

## 2. Почему нужен отдельный модуль

Данные из ЛИРА обычно приходят в виде таблиц: усилия, перемещения, реакции, результаты проверок, назначенные сечения. Для машинного обучения эти таблицы нужно привести к единому формату.

Отдельный модуль нужен, чтобы:

1. не смешивать расчётную логику и ML;
2. повторяемо импортировать новые расчёты;
3. проверять единицы измерения;
4. хранить версию ЛИРА, СП и модели;
5. исключать грязные или неполные кейсы;
6. строить одинаковые признаки для обучения и прогнозирования.

---

## 4. Основные сущности данных

### 4.1. `case_id`

Уникальный идентификатор одного расчётного случая:

```text
геометрия + материал + нагрузки + сочетания + сортамент + редакция норм
```

### 4.2. `section_id`

Идентификатор профиля из сортамента.

### 4.3. `target_section_id`

Самое лёгкое единое сечение, которое прошло проверки в ЛИРА для данного `case_id`.

### 4.4. `utilization`

Максимальный процент/коэффициент использования по проверкам. Для ML лучше хранить и процент ЛИРА, и нормированный коэффициент:

```text
utilization_ratio = utilization_percent / 100
```

## 5. Рекомендуемая структура датасета

### 5.1. Таблица `cases`

Одна строка = один расчётный случай.

Признаки:

- `span_m`
- `eave_height_m`
- `ridge_height_m`
- `roof_slope_deg`
- `frame_spacing_m`
- `building_length_m`
- `support_left`
- `support_right`
- `steel_grade`
- `snow_value_kpa`
- `wind_value_kpa`
- `terrain_type`
- `roof_dead_kpa`
- `purlins_kpa`
- `roof_live_kpa`
- `temperature_enabled`
- `cranes_enabled`
- `ice_enabled`

Метка:

- `target_section_id`
- `target_mass_kg_m`
- `max_utilization`

### 5.2. Таблица `element_envelopes`

Одна строка = элемент + экстремальные усилия.

- `case_id`
- `element_role`
- `length_m`
- `N_min_kN`
- `N_max_kN`
- `Q_max_abs_kN`
- `M_max_abs_kNm`
- `uy_max_abs_mm`
- `governing_combo`

### 5.3. Таблица `candidate_section_results`

Одна строка = расчёт одного `case_id` с одним кандидатом `section_id`.

- `case_id`
- `section_id`
- `mass_kg_m`
- `max_utilization`
- `pass_status`
- `governing_check`
- `governing_element`
- `governing_combo`

Эта таблица особенно полезна для обучения ранжирующей модели.

---

## 6. Разделение данных

Нельзя делать только случайное разделение строк, если в базе много похожих моделей. Рекомендуется:

1. Группировать по геометрии.
2. Отделять часть диапазонов пролёта/высоты в test.
3. Проверять extrapolation-случаи отдельно.
4. Хранить `split_id`: train/val/test.

---

## 7. Базовая модель

Для первой версии использовать простую baseline-модель:

1. RandomForest / Gradient Boosting / MLP для классификации сечения.
2. Метрика top-1 и top-5 accuracy.
3. Отдельная метрика безопасности: сколько раз модель предложила сечение легче безопасного.

Главная целевая метрика для инженерной безопасности:

```text
unsafe_rate = count(predicted section fails verification) / count(all predictions)
```

Для отчёта магистерской полезно сравнивать:

1. эталон ЛИРА;
2. прогноз нейросети;
3. ближайшее более тяжёлое безопасное сечение;
4. экономию времени подбора.

---

## 8. Требования к качеству данных

Кейс исключается из обучения, если:

1. нет геометрии;
2. нет выбранного сечения;
3. нет редакции СП;
4. нет результата pass/fail;
5. не указаны единицы;
6. есть противоречие между выбранным сечением и проверками;
7. расчёт ЛИРА завершился ошибкой;
8. одна и та же модель имеет разные метки без причины.

---

## 9. Безопасность применения нейросети

Нейросеть не должна быть единственным основанием для проектного решения.

Правильный pipeline:

```text
нейросеть предсказала top-5 сечений
        ↓
программа проверяет кандидатов
        ↓
если кандидат не проходит — берёт следующий
        ↓
если top-5 не проходят — запускает классический перебор
        ↓
отчёт показывает и прогноз, и проверку
```
